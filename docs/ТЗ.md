# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## Проект: WEB-AGENT

---

# 1. Общая концепция проекта

## 1.1 Назначение

WEB-AGENT — кроссплатформенное клиентское приложение, работающее в фоновом режиме и выполняющее команды, полученные от удалённого сервера по HTTP/HTTPS.

Система состоит из:

* Клиентского агента
* Серверной части (минимальная реализация или mock-сервер)
* WEB API для взаимодействия

---

# 2. Архитектура системы

## 2.1 Общая схема взаимодействия

```
[ Агент ] <--HTTP/HTTPS--> [ Сервер ]
     |                         |
     |---- регистрация ------>|
     |<--- Access Code -------|
     |---- запрос задачи ---->|
     |<--- задача + SessionID |
     |---- результат -------->|
     |<--- подтверждение -----|
```

---

## 2.2 Основные модули агента

Агент должен быть модульным:

1. **Agent** — главный управляющий класс
2. **HttpClient** — работа с HTTP/HTTPS
3. **SessionManager** — управление UID и Access Code
4. **TaskManager** — управление задачами
5. **TaskExecutor** — выполнение команд
6. **ConfigManager** — работа с конфигурацией
7. **Logger** — журналирование

Архитектура должна быть простой (без избыточных паттернов).

---

# 3. Функциональные требования

## 3.1 Регистрация агента

Агент должен:

* Иметь уникальный UID и descr (указаны в config)
* Отправлять UID и descr серверу
* Получать Access Code
* Обрабатывать отказ в регистрации

---

## 3.2 Получение задания

Агент должен:

* Периодически отправлять запрос:

  * UID
  * descr
  * Access Code

* Получать:

  * код задачи (task_code)
  * параметры (options)
  * номер сессии (session_id)
  * статус (status)

Если задания нет (код 0) — ожидание до следующего запроса.

---

## 3.3 Типы задач

Минимально поддерживаемые:

1. Выполнение системной команды
2. Запуск программы
3. Передача файла на сервер

---

## 3.4 Выполнение задачи

Агент должен:

* Запустить процесс
* Дождаться завершения
* Получить exit code
* Зафиксировать результат
* Переместить файлы в папку результатов (если требуется)

---

## 3.5 Передача результатов

Агент передаёт (через multipart/form-data):

* result_code (Код выполнения)
* result JSON строка (внутри нее передаются UID, Access Code, Session ID, лог/сообщение о выполнении, кол-во файлов)
* Файлы (если есть, отдельными частями multipart)

---

## 3.6 Обработка ошибок

* При недоступности сервера — увеличение интервала запросов
* При ошибке выполнения — передача кода ошибки
* Все ошибки фиксируются в логах

---

# 4. Нефункциональные требования

* Кроссплатформенность (Windows / Linux / macOS)
* Низкое потребление CPU и RAM
* Работа в фоновом режиме
* Минимальные зависимости
* Читаемый код
* Возможность сборки через CMake
